# 台北 e 大學習輔助程式：專案規格與分階段計畫

版本：v0.1 規格基線  
日期：2026-07-10

## 1. 專案定位

建立 Windows 桌面 GUI，協助使用者操作台北 e 大的學習紀錄、依序上課、選課及測驗填答流程。

「無障礙」是本專案的設計理念：減少重複點擊、集中顯示資訊、簡化課程選取及題目回填。此說明寫入 README，不另外建立無障礙功能模組。

## 2. 已確認的使用邊界

- 不設計帳號密碼欄位，不保存或自動輸入帳密。
- 程式啟動電腦已安裝的正式版 Google Chrome。
- 使用專用且持久化的 Chrome 使用者資料夾；第一次由使用者手動登入，後續沿用登入狀態。
- Chrome 與 GUI 分開顯示，不使用內嵌瀏覽器。
- Chrome 以最大化、原生 viewport 開啟，不沿用舊程式的 `1024×768` popup。
- 一次只進行一門課程，不同時多開課程。
- 不偽造學習紀錄，不以固定計時器宣告課程完成。
- 完成判定優先採用平台顯示的完成狀態；時數規則在實測確認前不得硬編碼。

## 3. 技術基線

- Python 3.12+
- PySide6
- Playwright，不使用 Selenium
- 正式 Chrome：`channel="chrome"`
- Persistent Context 保存登入狀態
- `no_viewport=True` 搭配 Chrome 最大化
- MVC 或相同程度的模組化分層
- Selector Manager 集中管理頁面元素
- 使用 locator、事件及頁面狀態等待，不用固定 `sleep` 作為主要流程控制
- logging、錯誤重試、逾時處理、畫面狀態與診斷資訊

## 4. GUI 資訊架構

建議採側邊導覽或分頁：

1. 儀表板
2. 學習紀錄／上課
3. 選課
4. 答題
5. 執行日誌／設定

### 4.1 儀表板

- Chrome 連線狀態
- 台北 e 大登入狀態
- 開啟 Chrome
- 重新偵測登入
- 進入學習紀錄
- 最近一次掃描時間與錯誤提示

### 4.2 學習紀錄／上課

課程表格欄位：

- 勾選
- 課程名稱
- 完成／未完成
- 平台原始認證時數文字
- 已上課時數
- 經驗證後的所需時數
- 剩餘時數
- 執行狀態

操作規則：

- 掃描後區分完成與未完成。
- 未完成課程預設全部勾選。
- 完成課程灰色顯示且預設不勾選。
- 使用者可取消不想上的課程。
- 按「開始上課」後依畫面順序逐門執行。
- 保留暫停、跳過、停止及手動下一門。
- 平台確認完成後，自動進入下一門已勾選課程。
- 無法可靠判定完成時停止自動前進並顯示原因，不猜測完成。

### 4.3 選課

- 可新增、刪除、排序及保存多個常用關鍵字。
- 一次批次搜尋全部關鍵字。
- 結果顯示課程名稱、時數、課程 ID、可否報名及目前狀態。
- 搜尋結果預設全部勾選。
- 使用者取消不要的課程。
- 按一次「開始報名」後依序處理。
- 每門課保留成功、已報名、不可報名及失敗原因。
- 不重複報名同一課程。

### 4.4 答題

固定流程：

1. 偵測目前測驗頁。
2. 擷取題目、單選／複選類型與選項順序。
3. 組合固定提示詞及題目純文字。
4. 直接複製到剪貼簿，不再詢問使用者。
5. GUI 顯示「已複製 N 題，請貼到 AI」。
6. 使用者將內容貼到 AI，再複製 AI 回傳結果。
7. 使用者把回傳結果貼入 GUI，或按「從剪貼簿讀取」。
8. GUI 驗證格式、題數、題號、選項範圍及單複選規則。
9. 驗證成功後直接填答並送出，不再詢問使用者。
10. 如果還有下一頁，自動擷取下一頁題目並複製到剪貼簿。
11. 驗證失敗時禁止送出，指出錯誤題號及原因。

### 4.5 GUI 優先開發與強制版面規則

這些項目屬於階段 1 的必要功能，不得延後到最後才美化：

- 先建立完整 GUI 骨架、導覽、內容區、狀態列及固定操作區，再接入各功能模組。
- 主視窗放大、縮小、最大化及還原時，元件必須隨版面重新配置。
- 禁止使用大量固定寬高或絕對座標排版；使用 Qt Layout、合理的 minimum size 及 size policy。
- 啟用 Qt High DPI 支援，至少測試 Windows 顯示比例 100%、125% 及 150%。
- 至少測試 1280×720、1366×768、1920×1080 三種解析度。
- 按鈕必須保留完整文字，不得因視窗縮小或顯示比例改變而只剩圖示、缺字或整段消失。
- 按鈕文字過長時優先調整 layout、minimum width 或按鈕排列，不以裁切文字處理。
- 主要操作按鈕使用一致的最小高度；停用狀態仍須清楚看得見文字。
- 內容可能超出畫面的頁面必須使用可捲動容器；底部主要操作列保持可見。
- 搜尋結果與選課清單不得使用不足以顯示一列文字的固定高度。
- 搜尋結果與選課清單出現多筆資料時，垂直捲軸必須可見、可拖曳且不遮住文字。
- 清單欄寬須支援自動延展或手動調整；長課程名稱不可被捲軸覆蓋。
- 表格至少保證目前選取列、課程名稱、狀態與主要操作欄可辨識。
- 搜尋結果不得只顯示第一筆；捲動後的列不得出現空白、重疊或文字消失。
- 切換頁面或重新整理資料後，應合理保留選取狀態與捲動位置。
- 文字與背景須維持足夠對比；一般、hover、selected、disabled 狀態都必須可讀。
- 所有頁面都要有載入中、空資料、成功及錯誤狀態，不以空白畫面代表狀態。

GUI 開發分成兩次：第一階段先完成穩定、不跑版且可讀的設計系統；所有功能完成後，再進行顏色、間距、圖示及細節精修。

## 5. AI 題目交換格式

### 5.1 固定提示詞

```text
請根據以下題目與選項作答。

只回傳一行答案，不要解釋、不要 Markdown、不要程式碼區塊。
格式必須完全如下：
[[ANSWERS]]1=A;2=BD;3=C[[/ANSWERS]]

規則：
1. 單選題填一個字母。
2. 複選題將答案字母直接相連，例如 BD。
3. 不得省略題號。
4. 不確定時仍選擇最可能的答案。
```

### 5.2 題目文字格式

```text
Q1 [單選]
題目內容
A. 選項一
B. 選項二
C. 選項三

Q2 [複選]
題目內容
A. 選項一
B. 選項二
C. 選項三
D. 選項四
```

### 5.3 GUI 解析規則

- 優先擷取 `[[ANSWERS]]` 與 `[[/ANSWERS]]` 之間的內容。
- 容許前後空白、換行、全形分號及小寫字母，解析後統一正規化。
- 題號必須與目前頁面完全一致。
- radio 單選題只能有一個答案字母。
- checkbox 複選題可有一個或多個答案字母。
- 答案字母不得超過該題選項數量。
- 任一題失敗即整頁不送出，避免部分錯位。

## 6. 模組拆分

### 核心層

- `BrowserManager`：啟動／關閉正式 Chrome、Persistent Context、分頁管理。
- `SessionDetector`：辨識未登入、已登入、登入逾時及錯誤頁。
- `PageRouter`：辨識首頁、學習紀錄、課程、教材、選課與測驗頁。
- `SelectorManager`：集中保存多組 selector 與備援策略。

### 功能層

- `LearningRecordScanner`：掃描課程與時數欄位。
- `CourseQueueService`：勾選清單、依序上課、自動下一門及中止恢復。
- `CourseNavigator`：進入課程、辨識多層頁面與教材入口。
- `CourseSearchService`：多關鍵字搜尋、去重與結果彙整。
- `EnrollmentService`：依序報名與結果確認。
- `QuizExtractor`：擷取題目、選項及單複選類型。
- `QuizPromptBuilder`：組合固定提示詞與題目文字。
- `AnswerParser`：解析並驗證 `[[ANSWERS]]` 格式。
- `QuizFiller`：依 input／label 關聯填答並送出。

### 支援層

- `SettingsRepository`：常用關鍵字、Chrome profile 路徑及 GUI 設定。
- `RunStateRepository`：課程佇列與中斷恢復狀態。
- `AppLogger`：檔案日誌與 GUI 日誌。
- `ErrorPresenter`：將技術錯誤轉成使用者可理解的訊息。

## 7. 既有 Console 程式的移植原則

來源：`https://tools.kumozoe.com/console.html`

### 可作為基線

- 台北 e 大上課 v2.2.9 的表格掃描、表頭辨識、時數文字解析及 `openNode`／教材入口辨識策略。
- 台北 E 大答題幫手的 `.que`、`.qtext` 及 radio／checkbox 題型概念。

### 不直接搬用

- 固定 `1024×768` popup。
- 大量 `setInterval`、固定延遲及以時間到期推定完成。
- 未經實測即固定使用「認證時數 × 0.5」。
- 額外增加 3 分鐘的補償規則。
- 依 `.answer div, .answer label` 取得選項造成的重複內容。
- `window.go()` 在 0.4 秒後直接按下一頁的舊方式。

Python 版應以實際 input 與 label 關聯建立選項順序，並先完成整頁答案驗證，再直接填答送出。

## 8. 分階段實作與驗收

### 階段 1：GUI 骨架與 Chrome 登入狀態

交付：

- PySide6 主視窗及五個功能頁。
- 正式 Chrome 最大化啟動。
- 專用 profile 可保存使用者手動登入狀態。
- 可辨識已登入、未登入與登入逾時。
- 建立共用的字體、顏色、按鈕、表格、捲軸及狀態元件樣式。
- 搜尋結果與選課清單先使用模擬資料驗證多筆顯示與捲動。
- 完成視窗縮放、Windows 顯示比例及低解析度適配。

驗收：

- 重新開啟程式後仍可沿用登入狀態。
- 頁面不因固定視窗尺寸遮住底部內容。
- 視窗放大、縮小、最大化及還原後，按鈕文字不得消失或被截斷。
- Windows 100%、125%、150% 顯示比例下，主要文字與按鈕均可讀。
- 搜尋及選課清單載入至少 30 筆模擬資料時，文字可見、捲軸可用且不遮住內容。
- 1280×720 下仍可操作所有主要按鈕；超出內容必須能捲動到達。

### 階段 2：學習紀錄掃描

交付：

- 掃描完成／未完成課程。
- 保存原始時數文字與正規化秒數。
- 顯示已修、所需與剩餘時數。
- 記錄無法解析的列及實際 HTML 特徵。

驗收：至少使用多種課程列結構交叉檢查，GUI 數值與平台畫面一致。

### 階段 3：依序上課

交付：

- 未完成課程預設全選。
- 依序進入每門課程及教材。
- 完成後自動下一門。
- 暫停、跳過、停止及錯誤恢復。

驗收：不同教材類型均能正確辨識入口；無法判定完成時不會錯誤跳課。

### 階段 4：選課

交付：

- 常用關鍵字持久化。
- 批次搜尋、去重、預設全選。
- 使用者取消後依序報名。
- 每筆結果與錯誤原因可追蹤。

驗收：已報名或不可報名課程不會被重複送出；報名結果可在平台確認。

### 階段 5：答題

交付：

- 擷取題目後自動複製提示詞與純文字。
- 解析固定 AI 回傳格式。
- 驗證成功後直接填答並送出。
- 多頁測驗自動擷取下一頁。

驗收：單選、複選、多頁、選項重複文字及格式錯誤均有測試；錯誤答案格式不得觸發送出。

### 階段 6：整合、測試與打包

交付：

- 單元測試、解析器 fixture、整合測試及執行日誌。
- README：安裝、首次登入、各模組操作、無障礙設計理念及使用邊界。
- Windows EXE 打包流程。

驗收：乾淨環境可啟動；缺少 Chrome、登入逾時、網路失敗與 selector 失效均有明確訊息。

## 9. 開發管理規則

- 一次只實作一個階段，不提前整合後續功能。
- GUI 骨架、縮放適配、文字可見性及清單捲動列為最高優先，不得等功能完成後才補救。
- 每階段開始前先列出要修改的檔案與驗收案例。
- 每階段完成後實際操作台北 e 大驗證，不只做靜態檢查。
- selector 失效時先記錄頁面類型與元素特徵，再更新 Selector Manager。
- 所有失敗嘗試保留課程、頁面、selector、錯誤訊息及判定原因。
- 未完成當階段驗收前，不打包 EXE。
